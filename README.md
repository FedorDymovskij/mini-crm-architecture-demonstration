# MINI CRM ARCHITECTURE DEMONSTRATION

Моя мини-CRM-система, впервые использующая сразу 3 архитектурных подхода: Feature-Sliced Design, DDD и Hexagonal
architecture.

---

## Функционал

- Регистрация и вход в аккаунт
- Аутентификация и авторизация за счет ротации *Refresh* и *Access* токенов
- Блокировка и разблокировка аккаунтов
- Методы для назначения пользователей модераторами и обычными пользователями
- Возобновление сессии

---

## Технологии

- Java 17
- Spring (**Security**, Boot, Web, Data JPA, Validation)
- Hibernate
- Flyway
- jjwt
- PostgreSQL
- Docker (Dockerfile, docker-compose)

---

## Архитектурные подходы и паттерны

- Feature-Sliced Design (FSD)
- Domain-Driven Design (DDD)
- Hexagonal architecture

---

## Особенности

- Сложная система аутентификации и авторизации за счет ротации *Refresh* и *Access* токенов
- Ротация *Refresh* и *Access* токенов при обращении к специальному эндпоинту
- Хранение состояния *Refresh* токенов в базе данных
- Наличие времени действия и статуса у *Refresh* токенов
- Валидация просроченных, заблокированных и обновленных *Refresh* токенов
- Обновление времени аутентификации при стабильном посещении ресурсов пользователем
- Защита от CSRF-атак (HttpOnly=true, Secure=true, SameSite=Strict)
- Валидация и защита *Access* токенов за счет секретного ключа для их подписи
- 3 заранее созданных пользователя с одноименными ролями для тестирования заготовленных сценариев:
    - username: **user**, password: **user**
    - username: **moderator**, password: **moderator**
    - username: **admin**, password: **admin**

## Установка

1. Клонируйте репозиторий:

```bash
git clone https://github.com/FedorDymovskij/crm-DDD

```

2. Перейдите в папку проекта:

```bash
cd .../crm-DDD

```

3. Установите JDK 17 (или выше)
4. Установите Apache Maven и скомпилируйте проект (либо используйте заранее скомпилированный проект):

```bash
./mvnw clean package -DskipTests 

```

5. Настройте файл `.env` (детали указаны внутри файла)
6. Соберите все контейнеры и запустите их:

```bash
docker compose up --build

```

#### Если вы используете IDE (например, IntelliJ IDEA), вы можете запустить базу данных отдельно

1. Поднимите конфигурацию `docker-compose-database.yml`:

```bash
docker compose -f ./docker-compose-database.yml up --build

```

2. Запустите основной проект из вашей IDE.

## Взаимодействие

### Эндпоинты:

| Метод    | URI                                  | Требования к запросу                                           | Описание                                                                 |
|----------|--------------------------------------|----------------------------------------------------------------|--------------------------------------------------------------------------|
| **POST** | `/register`                          | JSON: `{"username": "...", "password": "..."}`                 | Регистрация нового аккаунта (роль по умолчанию — **USER**)               |
| **POST** | `/login`                             | JSON: `{"username": "...", "password": "..."}`                 | Вход в аккаунт. Выдача *Refresh* токена в куке, *Access* токена — в JSON |
| **POST** | `/refresh`                           | Валидный *Refresh* токен в куке                                | Валидация и выдача новой пары *Refresh* и *Access* токенов               |
| **POST** | `/accounts/{id}/block`               | `Authorization: Bearer <Access>`, роль *ADMIN* или *MODERATOR* | Блокировка аккаунта с учетом бизнес-правил                               |
| **POST** | `/accounts/{id}/unblock`             | `Authorization: Bearer <Access>`, роль *ADMIN* или *MODERATOR* | Разблокировка аккаунта с учетом бизнес-правил                            |
| **POST** | `/accounts/{id}/assign-as-user`      | `Authorization: Bearer <Access>`, роль *ADMIN*                 | Назначение аккаунту роли *USER*                                          |
| **POST** | `/accounts/{id}/assign-as-moderator` | `Authorization: Bearer <Access>`, роль *ADMIN*                 | Назначение аккаунту роли *MODERATOR*                                     |
| **GET**  | `/accounts`                          | `Authorization: Bearer <Access>`, любая роль                   | Просмотр всех аккаунтов                                                  |
| **GET**  | `/accounts/{id}`                     | `Authorization: Bearer <Access>`, любая роль                   | Просмотр аккаунта по его айди                                            |

#### Подсказка по статус-кодам

* `200 OK` — Успешно.
* `400 Bad Request` — Плохие входные данные.
* `401 Unauthorized` — *Access* токен недействителен или отсутствует.
* `403 Forbidden` — Недостаточно прав.

## Дополнительно

Этот проект написан на основе моего предыдущего репозитория — `security-demonstration` — и является его логическим
продолжением.
Старая версия служила демонстрацией принципа работы *Refresh* и *Access* токенов для аутентификации и авторизации. В
`crm-DDD` архитектура была полностью переписана, что позволяет легко расширять проект, добавлять новые сущности,
бизнес-правила и контексты.
Было добавлено 4 новые функции, но при этом удалены все **GET**-эндпоинты. На текущий момент проект **НЕ ЗАКОНЧЕН**, в
нем могут встречаться несостыковки и слабые места. В связи с жизненными обстоятельствами я выгрузил проект в текущем
состоянии для последующей доработки. В будущем функционал будет расширен, архитектура доработана, а инфраструктура
оптимизирована.
